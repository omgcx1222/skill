# vibe-coding

**不要妄下结论。不要刻意掩饰困惑。要坦诚地面对权衡取舍。**

在开始之前:

- 倾向于编写清晰易读的代码，避免不必要的复杂性。
- 对于任何不明确、不了解的需求、逻辑，都需要先向开发者提问确认，不要默默替开发者做出选择。
- 对于变量的命名和复杂的逻辑，必须添加中文注释。
- 在开始生成代码前，先充分阅读理解当前需求相关的代码结构。
- 如果开发者指定了需求的md文档，又提出了偏离md文档的新需求，在代码生成完成后同步修改掉md文档的内容，并通知开发者。

## 1.简单至上

**用最少的代码解决问题。不要进行任何推测。**

- 不要开发需求之外的功能。
- 对于不可能出现的情况，不进行错误处理。
- 如果你写了 200 行，而 50 行就可以写完，那就重写。

## 2.精准修改

**只改动必须改的地方。只清理自己造成的混乱。**

修改现有代码时:

- 除非另有说明，否则不要修改或者删除原代码中相邻的代码、注释、控制台输出或格式。
- 不要重构没有问题的代码。
- 对于已经重复性的代码，优先考虑封装复用，避免重复编写。
- 保持与现有代码的风格一致，如果发现无关代码，请指出来，不要默默删除。
- 如果你创建了新的单个文件或者之前的代码废弃，删除因为**你的修改**而不再使用的导入、变量、方法。

如果当前的需求已经存在封装了的方法，请直接复用，如果无法满足当前的需求，先向开发者确认：

- 修改当前方法（但不要影响原有逻辑，因为这个方法可能在其它地方被调用了）。
- 直接创建新的方法。

测试标准：每一行改动的代码，都应能直接追溯到用户的需求。

## 3.目标驱动执行

**定义成功标准。循环执行，直到验证通过。**

将任务转化为可验证的目标：

- “增加校验” → “为非法输入编写测试，然后让测试通过”
- “修复这个 bug” → “写一个能复现该 bug 的测试，然后让测试通过”
- “重构 X” → “确保重构前后的测试都能通过”

对于多步骤任务，给出简要计划:

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

明确的成功标准能让你独立循环迭代。而模糊的标准（“只要能行就行”）则需要不断澄清。
